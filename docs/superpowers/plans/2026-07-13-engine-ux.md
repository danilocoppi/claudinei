# SP-C — UX por engine (frontend) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development para as tasks de BACKEND. As tasks de FRONTEND-VISUAL são executadas na conversa principal com a skill frontend-design (screenshots/crítica), conforme diretriz do usuário ("use sempre o mcp de frontend design qnd for mexer na parte web").

**Goal:** Escolher a engine ao iniciar um terminal, mostrar o ícone da engine em operação, e expor model/effort/permission/slash por engine — tudo dirigido pelas `capabilities` do backend (sem hardcode de engine no frontend).

**Architecture:** Backend expõe `GET /api/engines` (metadados + capabilities por engine registrada) e ganha `setEffort` na EngineSession. Frontend carrega essa lista no boot e monta seletor de engine, dropdowns e autocomplete a partir dela. STT/anexos intactos.

**Tech Stack:** Fastify 5 + vitest (backend); React 18 + zustand + react-i18next + vitest/testing-library (frontend); chrome-devtools MCP para validação visual.

## Global Constraints

- Spec: `docs/superpowers/specs/2026-07-13-engine-ux-design.md`.
- **Extensível:** o frontend NÃO enumera engines — renderiza o que `GET /api/engines` devolve. Uma 3ª engine aparece sozinha.
- **Claude intocado:** o fluxo do Claude (model/effort/permission/slash) permanece idêntico. Efeito do `/effort` do Claude continua sendo a mensagem de chat; o Codex usa só PATCH. A regra de "enviar /effort" NÃO é hardcode por engine — é: enviar a mensagem `/effort` sse o comando `effort` estiver na lista de slash da sessão (protocolo do Claude traz; curada do Codex não).
- **Sem rename `claudeSessionId`** (mantém estável; é o id de conversa da engine).
- Nomes/labels da UI em inglês; i18n en/pt-BR/es shape idêntico.
- Commits com trailer `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- Testes: `cd server && npm test` / `cd web && npm test`.
